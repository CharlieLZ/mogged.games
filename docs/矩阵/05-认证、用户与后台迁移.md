# 认证、用户与后台迁移

## 这块负责什么

这一层决定新项目是否继续沿用：

- Better Auth 登录体系
- 用户、Session、Account、Verification 表
- 积分、通知、任务历史
- 后台用户管理和权限模型

## 真相源

- `src/core/auth/*`
- `src/config/db/schema.ts`
- `src/app/api/auth/[...all]/route.ts`
- `src/app/[locale]/(auth)/*`
- `src/app/[locale]/(admin)/*`
- `src/shared/services/current-user.ts`
- `src/shared/services/settings.ts`
- `src/shared/services/rbac.ts`
- `src/shared/services/guest-viewer.ts`
- `src/shared/models/credit.ts`

## 默认建议

如果新项目仍然是“公开站点 + 登录后工作台 + 支付”，建议保留整套认证和用户骨架，只做删减，不要从零重建。

原因：

- 登录、回跳、Cookie、OAuth、邮件、会话续期已经踩过一轮坑
- 后台和用户工作台天然依赖这套 user/session/account 结构
- 迁移时保留数据模型，后续删模块的成本比重做认证低

## 标准迁移步骤

### 1. 先确认登录方式

决定是否保留：

- Email + password
- Magic link
- Google OAuth
- GitHub OAuth
- Google One Tap

对应 flag 全在 `src/config/index.ts`。

Google One Tap 额外检查：

- `GOOGLE_AUTH_ENABLED`
- `GOOGLE_ONE_TAP_ENABLED`
- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- OAuth consent screen 和 callback 域名
- 前端 availability 不能直接读 server-only env

### 2. 绑定 Auth 域名与 Secret

关键值：

- `AUTH_SECRET`
- `AUTH_URL`
- `NEXT_PUBLIC_APP_URL`

如果域名不一致，要先解释为什么；否则默认应该一致。

### 3. 复查用户表是否仍匹配新业务

重点看这些表是否继续需要：

- `user`
- `session`
- `account`
- `verification`
- `user_acquisition`
- `user_notification_preference`
- `user_notification`

### 4. 决定后台是否整套保留

如果新项目不需要当前后台的全部能力，优先做：

- 关闭入口
- 删掉路由
- 清理导航和文案

而不是保留一套空后台长期挂着。

### 5. 复查邮件触发链路

认证相关邮件包括：

- 密码重置
- magic link
- welcome email

新品牌上线前，要把邮件模板里的品牌名、域名、支持邮箱全部过一遍。

### 6. 复查注册赠送积分和 guest quota

如果新项目保留积分系统，登录不是只验证 session，还要验证权益链路：

- 新用户注册赠送积分需要 `INITIAL_CREDITS_ENABLED=true` 和 `INITIAL_CREDITS_AMOUNT` 同时在 Worker runtime 存在。
- 注册赠送积分写 `credit` 表，必须幂等，避免 OAuth hook 重试重复发。
- 登录后也要保留浏览器 guest 免费额度；guest quota 和账号 credits 是两套账。
- 购买积分、daily claim、注册赠送、退款恢复都在账号 credits ledger，不要和 guest quota 合并。

### 7. 复查管理员和证书入口

- 管理员账号用 `scripts/assign-role.ts --email=<email> --role=super_admin` 复查，不要靠前端头像或是否登录猜测。
- `Business Certificate` 这种复用 Settings shell 的页面，要确认 Settings 顶部导航 active 状态正确。
- 不保留证书能力的新项目，要同时关路由、侧栏入口、top nav 高亮和 pricing/upsell 文案。

## 验收

- 登录、注册、退出、重置密码都正常
- OAuth 回调域名正确
- 用户工作台不会跳回旧域名
- 后台权限边界与新业务一致
- 新注册用户能拿到注册赠送积分
- 登录用户仍能看到并使用浏览器 guest 免费额度
- 管理员账号权限和 Settings/Certificate 导航状态正确

## 常见坑

- 只换了前端域名，没换 `AUTH_URL`
- 关掉了某个 OAuth 按钮，但没关后端配置
- 新项目不需要证书/后台模块，却把旧入口保留到公开导航里
- 只配了 `INITIAL_CREDITS_AMOUNT`，忘了 `INITIAL_CREDITS_ENABLED`
- 登录后把匿名免费额度清掉，导致用户以为赠送积分或免费额度丢了
