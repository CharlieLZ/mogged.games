# 模板定位与复制策略

## 这份模板到底是什么

当前仓库更接近：

- 一个带公开站点的 AI 媒体产品基座
- 一个已经接上 Cloudflare、数据库、支付、用户、任务链路的运行中系统
- 一个适合继续孵化新域名产品的“业务模板”

当前仓库不应该被理解成：

- 通用博客模板
- 纯 Landing Page 模板
- 可无限插拔的无边界 SaaS 大杂烩

## 复制前先回答的三个问题

1. 新项目是否仍然需要两个工作台：图片和视频？
2. 新项目是否继续沿用当前的用户、支付、积分、任务模型？
3. 新项目是“换品牌复用业务骨架”，还是“复用骨架但删掉一半模块”？

如果这三个问题里有两个答案是否定的，就不要做“全量复制后再慢慢改”，而是应该先删模块再迁移品牌。

## 目录职责

| 目录 | 职责 |
| --- | --- |
| `src/app/` | 路由边界、metadata、API handlers |
| `src/shared/services/` | 业务编排，按功能聚合 provider 与 model |
| `src/shared/models/` | 数据访问与数据库记录 |
| `src/extensions/` | 外部 provider 适配层 |
| `src/shared/lib/` | 品牌、SEO、安全、限流、校验、工具函数 |
| `src/config/` | env、locale、站点级元数据、数据库 schema |

迁移文档必须沿着这个分层讲，不要把“改页面 copy”和“换支付回调地址”混成一件事。

## 推荐复制方式

### 方案 A：从当前仓库创建新 GitHub 仓库

适合：新项目和当前业务骨架相似度高。

标准步骤：

1. 从当前 `main` 创建新仓库或 template 仓库。
2. 第一轮只做品牌、域名、仓库、Cloudflare 项目切换。
3. 第二轮再删不需要的页面、provider、脚本、文案。

### 方案 B：先做删减版模板，再从删减版复制

适合：你已经知道下一批项目不会再用视频或某些增长模块。

标准步骤：

1. 在当前仓库先做一轮“删旧逻辑、收口模块、补文档”。
2. 把这个更窄的版本作为后续新项目的起点。

我更推荐方案 B。原因不是“更优雅”，而是它更少误迁移；你以后每开一个新站，少带一块没用系统，就少一块线上风险。

## 复制时的硬规则

- 根目录永远代表 `main`，新任务一律走 worktree。
- 不提交任何 `.env`、`.env-ref`、`.env.example`。
- 文档写当前真实系统，不写“将来可能做”。
- 先删旧逻辑和旧文档，再补最小可运行链路，最后才考虑抽象。

## 新项目初始化基线

创建新仓库后，先跑下面这组命令验证模板没坏：

```bash
pnpm install
pnpm verify:env
pnpm typecheck
pnpm test:unit
pnpm build
```

如果你打算走 Cloudflare：

```bash
pnpm cf:build
```

## 完成定义

只有同时满足下面几件事，才算“模板复制成功”：

- 新仓库里不再残留旧品牌、旧域名、旧仓库 URL、旧支持邮箱
- 关键模块开关和 env 配置都能说清楚是否启用
- `readurl.md`、README、迁移文档与实际路由一致
- 本地构建、测试、Cloudflare 构建都能过
