# Image Editor AI 模板化方案

## 文档目的

这份文档不是讲“怎么继续堆功能”，而是讲怎么把当前仓库收敛成一个可复用的 **同类 AI SaaS 骨架 + 产品包** 模板。

目标不是做一个什么都能装的万金油框架，而是做一套可以稳定复用到下一批 **同类 AI 媒体工作台项目** 的单仓库模板。

适用范围：

- 同类 AI 图片 / 视频 / 多媒体工作台
- 同类付费 SaaS，具备登录、积分、支付、任务、上传、通知、多语言、SEO 页面
- 同类“公开落地页 + 托管工作台 + 后台 / 用户中心”结构

不适用范围：

- 完全不同的信息站
- 纯内容站
- 没有任务、支付、上传、账号体系的轻工具站
- 希望一套模板同时覆盖十几种完全不同产品形态的场景

一句话原则：

**模板只保留能力，不保留产品结论。**

## 当前结论

当前仓库已经具备明显的模板化基础，但还没有真正完成“骨架”和“产品定义”的切割。

已经具备可复用骨架的部分：

- `src/shared/lib/brand.ts`
  已经承担了一部分品牌 token 归一化职责
- `src/config/index.ts`
  已经承担 runtime config 聚合职责
- `src/shared/services/manager-registry.ts`
  已经承担按配置注册能力的职责
- `src/extensions/*`
  已经具备第三方 provider 适配层雏形
- 当前 `app / services / models / extensions / lib` 分层方向是对的

当前仍然和具体产品强绑定的部分：

- `src/shared/lib/home-gallery.ts`
  直接绑定 Image Editor AI 的首页样片和远程素材
- `src/shared/blocks/generator/sample-media.ts`
  直接绑定当前公开工作流示例素材
- `src/config/website/public-page-metadata.ts`
  直接绑定当前公开 SEO 路由和更新时间
- `src/shared/lib/discoverable-pages.ts`
  直接绑定当前 discoverable 页面定义
- `src/config/db/schema.ts`
  schema namespace 仍然带有项目名痕迹
- `src/shared/services/guest-viewer.ts`
  guest cookie、global cache key 仍然带有项目名痕迹
- `src/config/ai-model-registry.ts`
  把 Seedance / KIE 默认链路和产品语义写死在全局

所以当前问题不是“架构完全不行”，而是：

**通用能力已经在，但产品定义散落在多个核心层文件里。**

## 模板化目标

目标是把这个仓库收敛成：

```text
一个单仓库
+ 一套稳定核心能力
+ 多个可切换产品包
+ 一个统一的创建新产品入口
```

这意味着后续扩新项目时，不应该再复制整仓库后手改，而应该：

1. 复用同一套核心能力
2. 新建一个 `src/products/<product-key>/`
3. 在产品目录内定义品牌、路由、文案、素材、provider 默认链路
4. 通过 manifest 装配到现有 app / services / blocks / seo / workspace 中

## 推荐目标结构

建议收敛后的结构如下：

```text
src/
  app/                        路由边界、metadata、API handlers
  core/                       真正通用的底座
  extensions/                 第三方 provider 适配
  shared/                     通用 block / lib / service / type
  products/
    image-editor-ai/
      manifest.ts
      brand.ts
      routes.ts
      features.ts
      providers.ts
      pricing.ts
      media.ts
      seo.ts
      copy/
        landing/
        pricing/
        emails/
        legal/
```

如果后续产品数量继续增长，再考虑把 `products/*` 拆到 workspace package；在当前阶段没有必要提前做 monorepo 复杂化。

## 分层原则

### 1. core 层

`src/core/` 只放真正跨产品稳定复用的底层能力：

- auth 基础设施
- db 连接与 repository 基础设施
- runtime config 读取框架
- task pipeline 基础设施
- storage 上传基础设施
- payment 基础设施
- i18n 基础设施
- observability 基础设施

core 层不允许知道：

- 当前品牌叫什么
- 默认公开域名是什么
- 当前有哪些公开工作流入口
- 当前默认用 Seedance 还是 KIE
- 首页长什么样

### 2. extensions 层

`src/extensions/` 只放第三方能力适配，不放产品选择结论：

- AI provider adapter
- payment provider adapter
- analytics provider adapter
- email provider adapter
- storage provider adapter
- customer service provider adapter

extensions 层只回答：

- “怎么接 Stripe / Creem / PayPal”
- “怎么接 Seedance / KIE / Fal / Replicate”

不回答：

- “当前产品默认启用谁”
- “当前产品免费用户走谁，付费用户走谁”

这些应上移到产品层配置。

### 3. shared 层

`src/shared/` 放跨产品可复用但不够底层的应用能力：

- 通用 UI block
- 通用 workspace shell
- 通用 SEO 工具
- 通用服务编排
- 通用模型访问
- 通用校验、限流、幂等、错误建模

shared 层可以依赖 core 和 extensions，但不应该继续埋产品名常量。

### 4. products 层

`src/products/<product-key>/` 才是具体产品定义所在：

- 品牌
- 域名
- support email
- repository URL
- 默认 locale
- 公开工作流入口
- landing block 组合
- 定价显示
- 默认 provider bundle
- 样片素材
- 邮件文案
- 法律页文案
- SEO 页面定义

一句话：

**所有“这个产品应该长成什么样”的问题，都归 products 层回答。**

## 产品包最小定义

每个产品包至少要有一个统一 manifest，建议命名为：

`src/products/<product-key>/manifest.ts`

这个 manifest 只放“高层产品事实”，不要放具体渲染逻辑。

建议最小结构如下：

```ts
export type ProductManifest = {
  key: string;
  appName: string;
  appUrl: string;
  appDomain: string;
  supportEmail: string;
  repositoryUrl: string;
  defaultLocale: string;
  locales: string[];
  themeKey: string;
  publicWorkflows: string[];
  defaultImageMode?: string;
  defaultVideoMode?: string;
  enabledCapabilities: {
    imageGeneration: boolean;
    videoGeneration: boolean;
    freeTools: boolean;
    pricingPage: boolean;
    guestMode: boolean;
  };
};
```

manifest 只负责告诉系统“这是什么产品”，不负责“怎么渲染这个页面”。

## 推荐拆分顺序

### 阶段 1：抽 product manifest

先把所有品牌级事实集中起来。

第一批应该收口到 manifest 或品牌相关文件中的信息：

- 品牌名
- 默认公开域名
- support email
- repository URL
- 默认 locale
- 支持语言
- 默认主题 key
- 公开工作流入口集合
- guest 模式是否启用
- 是否有视频工作台
- 是否有图片工作台

本阶段目标不是“全部解耦完成”，而是先建立一个唯一可信产品入口。

验收标准：

- 新产品不需要全仓搜索替换品牌名
- 大部分品牌级 token 都从一个入口读出
- `brand.ts` 不再承担半通用半产品的混合职责

### 阶段 2：抽 route + seo manifest

当前 `ai-video-generator`、`ai-image-generator`、`text-to-video` 这些路径仍然散落在多个文件里，后续复制项目极易漏改。

第二阶段需要把以下信息统一收口：

- 路由根路径
- 公开 canonical 页面
- noindex 页面
- sitemap discoverable 页面
- 页面 last modified 定义
- workflow path 与 mode 的映射
- 页面 title / description / keywords 的默认装配入口

建议每个产品包包含：

- `routes.ts`
- `seo.ts`

例如：

```ts
export type ProductRoutes = {
  imageGeneratorRoot?: string;
  videoGeneratorRoot?: string;
  aiModeLandingPaths: Record<string, string>;
  publicPages: string[];
  noindexPages: string[];
};
```

迁移对象优先级：

- `src/config/website/public-page-metadata.ts`
- `src/shared/lib/discoverable-pages.ts`
- 与 metadata、sitemap、canonical 相关的散落常量

验收标准：

- 新项目改路径不再需要改多个 SEO 文件
- 路由结构变化只影响产品包，不影响核心服务

### 阶段 3：抽 provider bundle

这是模板化成败的关键阶段。

当前问题不是没有 adapter，而是：

- adapter 在 `extensions/`
- 但“当前产品默认用谁”的决定散落在全局配置和 registry 里

应该改成：

- `extensions/` 只定义 provider 能力和契约
- `products/<product-key>/providers.ts` 定义该产品默认启用的 provider bundle

建议 provider bundle 回答这些问题：

- 图片生成默认 provider 是谁
- 视频生成默认 provider 是谁
- guest 用户走谁
- paid 用户走谁
- fallback 顺序是什么
- 哪些模型对外公开，哪些只内部启用
- 哪些 scene 在当前产品中可见

建议结构：

```ts
export type ProductProviderBundle = {
  image: {
    defaultProvider: string;
    guestProvider?: string;
    defaultSceneModels: Record<string, string>;
  };
  video: {
    defaultProvider: string;
    fallbackProviders: string[];
    defaultSceneModels: Record<string, string>;
  };
};
```

迁移对象优先级：

- `src/config/ai-model-registry.ts`
- 与 Seedance / KIE 默认链路绑定过深的 service 配置
- 与 scene -> model -> provider 绑定相关的全局常量

验收标准：

- 新项目切换默认 provider 不需要动全局 registry 逻辑
- 当前产品的 provider 选择从“代码事实”变成“产品配置事实”

### 阶段 4：抽 marketing / media pack

这一步的重点是把“可见内容层”彻底产品化。

应迁移到产品目录的内容包括：

- 首页 gallery
- sample media
- landing page block 组合
- hero / feature / CTA 文案
- pricing 展示文案
- mission 文案
- 邮件模板文案
- 法律页内容

建议新增：

- `media.ts`
- `features.ts`
- `pricing.ts`
- `copy/landing/*`
- `copy/emails/*`
- `copy/legal/*`

迁移对象优先级：

- `src/shared/lib/home-gallery.ts`
- `src/shared/blocks/generator/sample-media.ts`
- 首页和 pricing 页面中直接写死品牌语义的 block 数据
- 邮件模板中的产品描述文案

验收标准：

- 新项目替换首页样片不需要动 shared 层
- 新项目替换邮件文案不需要改核心 service

### 阶段 5：抽 create-product 脚本

前四阶段完成后，再做自动化脚手架才有意义。

否则会变成：

- 复制了一堆还没解耦干净的旧文件
- 再用脚本把问题自动复制一遍

create-product 的职责应该是：

1. 生成 `src/products/<product-key>/`
2. 生成最小 manifest、routes、seo、providers、pricing、media 文件
3. 生成多语言文案目录骨架
4. 注册到产品索引
5. 输出后续人工补充 checklist

不应该做的事：

- 自动复制一整套业务逻辑
- 自动复制一整套旧 landing 素材
- 自动写一堆并不真实的 SEO 文案

## 推荐文件职责调整

### 现有文件保留并瘦身

建议保留但瘦身的文件：

- `src/shared/lib/brand.ts`
  改成从当前 active product manifest 读取品牌 token
- `src/config/index.ts`
  保持 runtime config 聚合职责，但不要继续承担产品定义默认值的唯一来源
- `src/shared/services/manager-registry.ts`
  保持通用注册器，不绑定任何产品逻辑

### 现有文件迁移方向

建议迁移的文件职责：

- `src/shared/lib/home-gallery.ts`
  拆到 `src/products/<product-key>/media.ts`
- `src/shared/blocks/generator/sample-media.ts`
  拆到 `src/products/<product-key>/media.ts`
- `src/config/website/public-page-metadata.ts`
  拆到 `src/products/<product-key>/seo.ts`
- `src/shared/lib/discoverable-pages.ts`
  拆到 `src/products/<product-key>/seo.ts`
- `src/config/ai-model-registry.ts`
  拆成“通用模型注册规则”与“产品默认 bundle”
- `src/config/db/schema.ts`
  引入可配置 schema key，不继续写死项目名
- `src/shared/services/guest-viewer.ts`
  guest cookie name、cache key、global key 变成品牌无关或 product-key 驱动

## 底层命名治理

这是后续复制模板最容易漏的地方，必须单独处理。

### 1. schema 命名

当前数据库 schema 名称直接带仓库身份。

问题：

- 复制仓库后容易遗留旧 schema 名称
- 多产品部署时难区分真实租户边界
- 后续做部署自动化时容易出现误连

建议：

- schema 名称改为环境变量或产品 manifest 驱动
- 明确区分“代码仓库名”和“数据库命名空间”

建议形态：

```ts
const schemaKey = process.env.DB_SCHEMA_KEY || activeProduct.key;
export const siteSchema = pgSchema(schemaKey.replace(/-/g, '_'));
```

### 2. cookie / storage / cache key 命名

当前 guest cookie 与部分缓存全局变量带产品名。

问题：

- 新项目复制后极易残留旧产品痕迹
- 同域或近似域并存时容易让排障成本变高

建议：

- cookie name 统一走产品 key 驱动
- cache key、global singleton key 统一引入命名前缀构造器

例如：

```ts
const guestCookieName = `${activeProduct.key}_guest_id`;
```

### 3. 日志和监控命名

当前很多日志消息带有当前产品语义。

建议：

- 核心服务日志记录 capability / provider / scene / taskId
- 品牌信息只作为附加字段，不作为逻辑判断条件

## 路由与产品索引建议

为了让系统真正支持多产品扩展，建议引入一个统一产品索引：

`src/products/index.ts`

职责：

- 导出所有产品 manifest
- 决定当前 active product
- 为 app、seo、workspace、email、pricing 提供统一读取入口

建议最小接口：

```ts
export function getActiveProduct(): ProductDefinition
export function getProductByKey(key: string): ProductDefinition | null
export function listProducts(): ProductDefinition[]
```

active product 的选择方式，推荐优先级如下：

1. 显式环境变量指定
2. 部署域名映射
3. 默认产品回退

不要把产品判断逻辑散在几十个页面文件和 service 文件里。

## 对当前仓库最现实的实施边界

这个项目现在最适合被收敛成：

**同类 AI 媒体 SaaS 骨架**

不是：

- 通用 CMS 模板
- 通用工具站模板
- 任意 SaaS 模板

当前最现实的复用边界是：

- 登录 / 支付 / 积分 / AI 任务 / 上传 / 通知 / 多语言 / SEO
- 图片工作台
- 视频工作台
- 公开落地页
- 用户中心 / 后台

超出这个边界的产品，应该新开骨架，不要强行塞回这个模板。

## 实施顺序建议

建议严格按下面顺序推进，不要跳步：

1. 先补文档真相和边界定义
2. 再建立 `products/<product-key>/manifest.ts`
3. 再迁移 route + seo
4. 再迁移 provider bundle
5. 再迁移 media / copy / pricing
6. 最后才做 create-product 脚本

原因很简单：

- 没有唯一产品入口前，后面所有迁移都会继续散
- 没有 route / seo 收口前，新项目复制仍然容易漏索引页
- 没有 provider bundle 前，新项目只是换皮，不是真正换产品
- 没有 content / media 收口前，模板仍然会带着旧品牌味道

## 风险与反模式

### 不要做的事

- 不要一上来改 monorepo
- 不要先写 create-product 脚本
- 不要先做抽象接口大全
- 不要把所有内容都塞回 `shared/`
- 不要把产品差异继续藏在 env 默认值里
- 不要把“当前默认 provider”继续写在全局 registry 常量里

### 最常见失败方式

- 抽象过早，目录更复杂但耦合没变少
- 名义上有 `products/`，实际关键路由仍散落全局
- 核心层继续知道品牌名和默认域名
- provider adapter 和 provider selection 没分开
- 文案、素材、SEO 只换了一半，复制新项目后仍然残留旧品牌

## 完成标志

当下面这些条件成立时，才算真正进入“可模板化”状态：

- 新产品可以通过新增 `src/products/<product-key>/` 开始，而不是复制整个仓库
- 品牌、域名、路由、SEO、样片、邮件、法律文案都能从产品目录找到
- provider 默认链路可以在产品层切换
- shared / core / extensions 不再直接写死当前产品名
- 新项目初始化不再依赖全仓搜索替换旧品牌词
- create-product 脚本只生成产品包，不复制业务实现

## 推荐下一步

下一步不是直接大改，而是先做一轮最小设计落地：

1. 新建 `src/products/image-editor-ai/`
2. 只先抽 `manifest.ts`
3. 让 `brand.ts` 改为读取 active product manifest
4. 在不改业务流程的前提下，把品牌级事实先收口
5. 完成后再进入 route + seo 迁移

这样做的好处是：

- 改动最小
- 回滚最容易
- 可以快速验证产品层入口是否站得住

## 结论

这个仓库适合被收敛成模板，但模板边界必须明确：

它应该成为一套 **同类 AI SaaS 骨架 + 产品包**，而不是一个覆盖所有网站形态的通用框架。

真正的关键不是“抽更多公共函数”，而是把：

- 能力
- 产品事实
- 第三方适配
- 内容素材

这四层重新分开。

如果后续严格按本文顺序推进，就可以把当前 `Image Editor AI` 收敛为第一个产品包，并为下一批同类项目留出稳定扩展面。
