# 环境变量与运行时配置

## 这块负责什么

模板迁移时，最容易出事故的不是页面，而是：

- 本地 `.env` 能跑，但 Cloudflare 线上没值
- 只换了公开域名，没换 Auth / Webhook / 回调
- 某个 provider flag 开着，但 key 还是占位值

## 真相源

- `src/config/index.ts`
- `scripts/verify-env.ts`
- `wrangler.jsonc`
- `scripts/setup-cf-vars.mjs`
- `scripts/setup-cf-secrets.sh`
- `read-checking-list.md`
- Cloudflare Dashboard 的 Worker vars / secrets

## 配置分层

### 强制必填

- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_APP_NAME`
- `DATABASE_URL`
- `DATABASE_PROVIDER`
- `AUTH_SECRET`

### 强烈建议显式配置

- `AUTH_URL`
- `GOOGLE_ANALYTICS_ID`
- `PLAUSIBLE_DOMAIN`
- `PLAUSIBLE_SRC`

### 按模块启用

- Auth: `GOOGLE_*` `GITHUB_*`
- Payment: `STRIPE_*` `CREEM_*` `PAYPAL_*`
- Email: `RESEND_*` `ZEPTOMAIL_*`
- Storage: `R2_*` `S3_*`
- AI: `OPENROUTER_*` `VOLCENGINE_*` `APIXO_*` `APIMART_*` `EVOLINK_*` `KIE_*` `FAL_*` `REPLICATE_*`
- Growth: `CRISP_*` `TAWK_*` `AFFONSO_*` `PROMOTEKIT_*`
- Analytics/Ads: `GOOGLE_ANALYTICS_ID` `GOOGLE_ADS_*` `CLARITY_ID` `PLAUSIBLE_*`

## 标准迁移步骤

### 1. 先做一份 env 映射表

不要先改 `.env.example`。优先在文档或表格里整理：

- 变量名
- 是否必填
- 是否敏感
- 新项目是否启用
- 值来自哪里
- 应该放到本地 `.env`、Cloudflare `[vars]` 还是 `wrangler secret`

### 2. 把模块开关和 key 绑定

例如：

- `stripe_enabled=true` 但 `STRIPE_SECRET_KEY` 为空，是错误
- `google_auth_enabled=true` 但 OAuth client 未配置，是错误
- `initial_credits_amount=150` 但 `initial_credits_enabled` 不是 `true`，新用户注册不会送积分
- `crisp_enabled=false` 时，没有必要保留 Crisp 的占位值

### 3. 明确 public vars 和 secret 的边界

建议规则：

- 公开可见值放 `[vars]`
- 密钥类值放 `wrangler secret` 或 Cloudflare runtime secrets
- 本地 `.env` 只用于开发与脚本，不作为生产真相源
- 稳定、非敏感、生产必须的 Worker plain-text vars 要写入 `wrangler.jsonc`
- 新增 plain-text runtime key 时同步 `scripts/setup-cf-vars.mjs`
- 新增 secret key 时同步 `scripts/setup-cf-secrets.sh`

### 4. 复核 Worker runtime，而不是只看本地文件

生产事故里最常见的模式是：本地 `.env` 有值，但 Cloudflare Worker 没有完整 binding。

必须分开看：

- 本地 `.env`：本地脚本和本地开发用
- `wrangler.jsonc`：稳定 plain-text Worker vars 和 routes
- Cloudflare secrets：`DATABASE_URL`、`AUTH_SECRET`、provider secret、API token
- 数据库 `config` 表：非空小写 config 会覆盖 envConfigs
- 构建产物：不能带出 `.env` 或 secret

### 5. 跑 env 校验

```bash
pnpm verify:env
```

处理原则：

- `errors` 必须清零
- `warnings` 需要逐条确认是“有意关闭模块”还是“遗漏配置”

## Cloudflare 侧建议

### 放到 `[vars]` 的典型值

- `NEXT_PUBLIC_APP_URL`
- `NEXT_PUBLIC_APP_NAME`
- `AUTH_URL`
- `DATABASE_PROVIDER`
- `DB_SINGLETON_ENABLED`
- `DEFAULT_PAYMENT_PROVIDER`
- `STRIPE_ENABLED`
- `STRIPE_PUBLISHABLE_KEY`
- `INITIAL_CREDITS_ENABLED`
- `INITIAL_CREDITS_AMOUNT`
- `INITIAL_CREDITS_VALID_DAYS`
- `DAILY_CLAIM_CREDITS_AMOUNT`
- `GOOGLE_ANALYTICS_ID`
- `PLAUSIBLE_DOMAIN`
- `PLAUSIBLE_SRC`

### 放到 secrets 的典型值

- `AUTH_SECRET`
- `DATABASE_URL`
- `STRIPE_SECRET_KEY`
- `STRIPE_SIGNING_SECRET`
- `CREEM_API_KEY`
- `PAYPAL_CLIENT_SECRET`
- `RESEND_API_KEY`
- `ADMIN_NOTIFICATION_EMAIL`
- `CONTACT_NOTIFICATION_EMAIL`
- `R2_SECRET_KEY`
- `KIE_API_KEY_*`

说明：`ADMIN_NOTIFICATION_EMAIL` / `CONTACT_NOTIFICATION_EMAIL` 本身不是技术意义上的 secret，但属于运营收件人信息。是否放 secret 看团队习惯；关键是不要只配置在本地 `.env`，线上 Worker 必须能读到。

## 验收

- `pnpm verify:env` 没有 error
- `pnpm build` 和 `pnpm cf:build` 都能读取到必要配置
- 生产环境实际回调 URL、支付 URL、Auth URL 全部指向新域名
- `read-checking-list.md` 的 env/key 总盘点已同步
- Cloudflare Worker runtime 能看到新增 vars / secrets

## 常见坑

- 误以为 `wrangler.toml.example` 就是线上值
- 误把本地 `.env` 当生产配置
- 开着 provider 开关，但没填真实 key
- 忽略 `AUTH_URL` 与 `NEXT_PUBLIC_APP_URL` 不一致的 warning
- 只配 `INITIAL_CREDITS_AMOUNT`，忘了 `INITIAL_CREDITS_ENABLED`
- 只改 `scripts/setup-cf-vars.mjs`，忘了把稳定 vars 写进 `wrangler.jsonc`
- 只看 `NEXT_PUBLIC_EMAIL`，忘了 contact/support 表单真正用的是 contact/admin/support fallback
